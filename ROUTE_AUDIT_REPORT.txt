5. AUTHENTICATION FLOW
--------------------------------------------------------------------------------
1. INITIATION: User visits /login or /signup.
2. IDENTITY MAPPING:
   - Google SSO button triggers mock google user or logs in with Firebase pop-up.
   - Microsoft SSO button mock logs in user as admin@gmail.com.
   - Email/Password form triggers login or registration. Bypasses firebase checks for:
     * agent@gmail.com -> mocks agent user id.
     * admin@gmail.com -> mocks microsoft admin user.
3. SESSION MANAGEMENT:
   - App.tsx tracks auth updates and caches details in localStorage under `sentinel_mock_user`.
   - Logging out clears the local key and invokes signOut from Firebase auth.

7. ROUTE GUARDS & 8. REDIRECT LOGIC
--------------------------------------------------------------------------------
All logic is executed inside the AnimatedRoutes useEffect inside App.tsx:

FILE: App.tsx
FUNCTION: AnimatedRoutes -> useEffect()
CURRENT BEHAVIOR:
- IF user is logged in:
  - If location is /login or /signup, triggers setUser(null) to sign out user. Prevents back navigation loop trap.
  - If email === admin@gmail.com and location !== /admin-control-tower, redirects to /admin-control-tower.
  - If email === agent@gmail.com and location is not dashboard, redirects to /dashboard.
  - For standard users:
    * If onboarding is incomplete, redirects any requests for /dashboard or /admin-control-tower to /onboarding.
    * If onboarding is complete, redirects requests for /admin-control-tower to /dashboard.
- IF user is NOT logged in:
  - Redirects requests for /dashboard, /onboarding, or /admin-control-tower to /login.

10. ROUTING ISSUES & DETECTED PROBLEMS
--------------------------------------------------------------------------------
1. Merged Paths: /dashboard/sales-mgmt renders CRMView directly, and /dashboard/purchase
   renders ProcurementView directly. These are duplicate navigation targets for unified pages.
2. Route Isolation: If a standard user manually types /dashboard/ecc or other agent paths,
   they can access those components, as permissions: ['*'] is set in registry.ts. There is no
   explicit module-level role block in registry components, rely solely on sidebar filtering.
3. Onboarding Access: Even after onboarding is completed, standard users can navigate back 
   to /onboarding page manually, which is permitted.

--------------------------------------------------------------------------------
11. RECOMMENDED ROLE MAPPING
--------------------------------------------------------------------------------
- Standard users should be locked to permissions matching their onboarding modules list.
- Explicit components-level checks should prevent standard users from rendering agent-only
  pages under /dashboard/ecc etc. if accessed by manual URL input.
